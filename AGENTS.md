# AGENTS.md

## 适用范围
- 本文件作用于仓库根目录及全部子目录。
- 若子目录存在更深层 AGENTS.md，以更深层为准。

## Codex 后续开发规则
1. 先读文档再改代码：开始任何开发前，先同步阅读 `docs/00-09` 蓝图文档。
2. 先设计后实现：涉及跨模块改动时，先在 `docs/` 补充设计说明，再提交代码。
3. 小步提交：每个 PR 聚焦单一主题，避免“文档+大规模业务重构”混在一个 PR。
4. 变更可追溯：所有关键设计决策必须在 PR 描述中给出依据（文件路径/接口/数据模型）。
5. 禁止跳过验证：没有测试结果时，不得声称“已验证通过”。
6. 涉及 To B 机构流程时，必须同步说明“机构视角、学生视角、管理视角”三方影响。

## 代码风格要求
- 前端：TypeScript + React 函数组件，优先 hooks，避免巨型页面组件。
- 后端：NestJS 模块化结构，Controller 仅处理协议转换，业务逻辑放 Service。
- 数据访问：统一通过 Prisma，避免散落原生 SQL（除性能必要且有注释说明）。
- 命名：
  - 文件名 kebab-case；
  - 类型/类 PascalCase；
  - 变量/函数 camelCase；
  - 常量 UPPER_SNAKE_CASE。
- 错误处理：统一错误码与 message，避免前端硬编码后端异常文本。
- 注释：解释“为什么”，而不是重复“做了什么”。

## PR 要求
- 标题格式建议：`type(scope): summary`，例如 `feat(order): add refund resolve flow`。
- 描述至少包含：
  1. 变更内容
  2. 影响范围（前端/后端/数据库/部署）
  3. 测试结果（命令 + 结果）
  4. 回滚方案
  5. 是否需要数据迁移
- 涉及接口变更时：必须更新 `docs/06-api-design.md`。
- 涉及数据模型变更时：必须更新 `docs/05-data-model.md` 与 Prisma migration 说明。
- 涉及机构（To B）流程变更时：必须更新 `docs/01`、`docs/02`、`docs/03` 的对应章节。

## 测试和构建要求
- 前端改动至少执行：
  - `pnpm --filter web lint`
  - `pnpm --filter web test`
  - `pnpm --filter web build`
- 后端改动至少执行：
  - `pnpm --filter api lint`
  - `pnpm --filter api test`
  - `pnpm --filter api build`
- 涉及关键链路（登录/下单/支付/任务推进）时，必须补充 e2e 或脚本验证结果。
- 涉及机构链路（推单/代下单/协同交付）时，必须补充该链路验证结果。

## 本地开发与验证环境规范

### 一、技术栈说明
- 前端使用 Next.js。
- 后端使用 NestJS。
- 本地可通过 Docker / Docker Compose 启动依赖服务。
- GitHub 是代码协作中心。
- Trae 本地环境用于运行、调试和人工验收。
- Codex 不应假设代码只在云端运行，所有改动必须能在本地 Docker 环境中验证。

### 二、Codex 开发规则
1. 每次只处理一个 GitHub Issue。
2. 每个 Issue 创建一个独立分支。
3. 默认基于 `dev` 创建分支。
4. 默认 PR 目标分支为 `dev`。
5. 不允许直接修改 `main`。
6. 不允许一个 PR 同时处理多个无关任务。
7. 不允许无关重构。
8. 不允许修改与当前 Issue 无关的文件。
9. 不允许提交真实密钥、Token、数据库密码、服务器地址等敏感信息。
10. 如果发现文档与代码现状冲突，必须采用最小可行修改，并在 PR 中说明。

### 三、本地验证要求
每个开发 PR 完成后，Codex 必须在 PR 描述中提供建议验证步骤，包括但不限于：
1. 安装依赖命令。
2. 启动 Docker 服务命令。
3. 启动 NestJS 后端命令。
4. 启动 Next.js 前端命令。
5. lint 命令。
6. typecheck 命令。
7. test 命令。
8. build 命令。
9. 需要人工访问的页面路径。
10. 需要人工测试的核心功能点。

### 四、前端验证要求
涉及 Next.js 页面或组件的 PR，必须说明：
1. 影响的页面路径。
2. 需要测试的浏览器页面。
3. loading 状态。
4. empty 状态。
5. error 状态。
6. 表单校验状态。
7. 移动端适配情况。
8. 刷新页面后的状态。

### 五、后端验证要求
涉及 NestJS 接口的 PR，必须说明：
1. 影响的 Controller。
2. 影响的 Service。
3. 影响的 DTO。
4. 影响的数据库实体或 ORM 模型。
5. 需要测试的接口路径。
6. 鉴权要求。
7. 请求参数。
8. 返回结构。
9. 异常情况。

### 六、Docker 验证要求
涉及环境、数据库、部署或后端依赖的 PR，必须说明：
1. 是否需要 `docker compose up -d --build`。
2. 是否需要数据库迁移。
3. 是否需要 seed 初始化数据。
4. 是否新增环境变量。
5. 是否更新 `.env.example`。
6. 是否影响已有容器服务。

### 七、PR 描述要求
每个 PR 必须包含：
1. 对应 Issue。
2. 源分支。
3. 目标分支。
4. 本次目标。
5. 修改内容。
6. 修改文件列表。
7. 本地验证步骤。
8. 页面访问路径。
9. 接口测试方式。
10. Docker 验证方式。
11. 风险点。
12. 需要人工确认的问题。
13. 人工验收清单。

# Git 分支、Codex PR 与本地验证规范

## 一、基础分支规则

1. main 是正式上线分支，只允许合并经过验收的稳定代码。
2. dev 是开发集成分支，所有功能 PR 默认合并到 dev。
3. 不允许直接向 main 提交代码。
4. 不允许直接向 main 创建 PR，除非是明确的 release 或 hotfix 任务。
5. Codex 每次处理开发任务时，必须基于 dev 创建独立任务分支。
6. 每个 GitHub Issue 对应一个独立分支和一个独立 PR。

## 二、分支命名规则

1. 功能开发：feature/issue-编号-简短英文描述
2. Bug 修复：fix/issue-编号-简短英文描述
3. UI 优化：ui/issue-编号-简短英文描述
4. 文档调整：docs/issue-编号-简短英文描述
5. 配置或部署：chore/issue-编号-简短英文描述
6. 上线候选：release/v版本号
7. 紧急修复：hotfix/issue-编号-简短英文描述

示例：

- feature/issue-001-login-flow
- fix/issue-002-build-error
- ui/issue-003-dashboard-layout
- chore/issue-004-deployment-config
- docs/issue-005-update-readme
- release/v1.0.0

## 三、PR 目标分支规则

1. feature/*、fix/*、ui/*、docs/*、chore/* 分支默认 PR 到 dev。
2. release/* 分支 PR 到 main。
3. hotfix/* 可以从 main 拉出，修复后 PR 到 main，并同步回 dev。
4. 如果任务没有明确说明，Codex 必须默认 PR 到 dev。

## 十、合并规则

1. PR 必须先经过 Codex Review。
2. PR 必须通过本地或 CI 的构建检查。
3. PR 必须由人工验收核心功能后才能合并。
4. P0/P1 问题未解决前不得合并。
5. dev 阶段测试通过后，才能创建 release 分支。
6. release 分支验收通过后，才能合并 main。

## 十一、禁止事项

1. 禁止直接修改 main。
2. 禁止一个 PR 同时处理多个无关任务。
3. 禁止无关重构。
4. 禁止提交真实密钥、Token、数据库密码、服务器地址等敏感信息。
5. 禁止绕过 Review 直接合并。
6. 禁止在没有验证结果的情况下标记任务完成。

## 禁止事项
- 禁止在未确认影响的情况下直接修改生产相关支付逻辑。
- 禁止将密钥、证书、token、密码写入仓库。
- 禁止绕过鉴权与权限检查（包括临时注释 Guard 后直接提交）。
- 禁止在无迁移脚本的情况下变更数据库结构。
- 禁止修改文档与实现不一致后不更新文档。
- 禁止“顺手修复”与当前任务无关的业务逻辑。
- 禁止实现机构数据可跨机构越权访问。
