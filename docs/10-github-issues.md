# 10 GitHub Issues 拆分清单（P0 重排版）

> 目标调整：P0 不再以“页面骨架”为主，而是优先完成 **Client 端可用业务闭环**（登录、生成论文题目、生成开题报告、生成论文正文）。

## P0 总体原则
- 每个 Issue 必须交付“可用能力”，不是仅静态页面。
- 每个 Issue 必须带接口联调、错误处理、最小验收脚本。
- P0 阶段 To B（机构）仅保留不阻塞 Client 闭环的最小支持。

---

## [P0-CLIENT-01] 客户登录可用化（认证闭环）

**目标**
- 完成客户登录、鉴权态持久化、登出、鉴权失效处理。

**范围**
- 前端：登录页表单、校验、状态反馈、token 持久化、路由守卫。
- 后端：登录接口返回结构对齐、错误码对齐（如凭证错误、账号禁用）。

**验收标准**
- 正确账号可登录并进入客户工作台。
- token 过期后可正确跳转登录并提示。
- 错误凭证有明确错误提示，不泄漏敏感信息。

**建议 PR 标题**
- `feat(client-auth): deliver usable customer login flow`

---

## [P0-CLIENT-02] 客户生成论文题目（可提交+可查看）

**目标**
- 客户可以提交论文题目生成请求，并查看生成结果与历史。

**范围**
- 前端：题目生成表单（专业、方向、关键词、约束条件）、结果列表、重试。
- 后端：题目生成接口、任务入队/状态返回、历史查询。

**验收标准**
- 可提交一次有效请求并返回题目候选结果。
- 网络异常/生成失败可重试。
- 刷新后历史记录仍可查看。

**建议 PR 标题**
- `feat(client-topic): enable thesis topic generation workflow`

---

## [P0-CLIENT-03] 客户生成开题报告（可追踪状态）

**目标**
- 客户基于已确认题目生成开题报告，并可查看任务状态推进。

**范围**
- 前端：开题报告生成入口、状态展示（loading/queued/running/success/failed）、结果预览。
- 后端：开题报告生成任务接口、状态查询接口、失败原因结构化返回。

**验收标准**
- 基于题目可成功发起开题报告任务。
- 可见明确状态流转与失败原因。
- 成功后可查看报告摘要/内容。

**建议 PR 标题**
- `feat(client-proposal): deliver proposal generation and tracking`

---

## [P0-CLIENT-04] 客户生成论文正文（分阶段可交付）

**目标**
- 客户可发起正文生成任务，并按章节/阶段查看结果。

**范围**
- 前端：正文生成参数页、章节进度面板、失败重试入口。
- 后端：正文生成任务编排、阶段性状态与结果查询。

**验收标准**
- 可发起正文生成并获得可追踪任务 ID。
- 章节/阶段进度可视化，失败可重试。
- 结果可预览并进入下载流程。

**建议 PR 标题**
- `feat(client-thesis): implement thesis body generation workflow`

---

## [P0-CLIENT-05] Client 统一任务时间线（题目/开题/正文）

**目标**
- 将三类任务统一进客户可见时间线，形成端到端进度视图。

**范围**
- 前端：统一 timeline 组件、筛选（任务类型/状态）、详情抽屉。
- 后端：统一 timeline 聚合接口（仅客户本人数据）。

**验收标准**
- 同一订单下可查看题目/开题/正文全链路时间线。
- 状态含义一致，失败节点含可读原因。

**建议 PR 标题**
- `feat(client-task): unify customer timeline for core deliverables`

---

## [P0-CLIENT-06] Client 下载与交付验收最小闭环

**目标**
- 客户可下载题目/开题/正文交付物，具备基础验收动作。

**范围**
- 前端：下载列表、文件状态、失败提示。
- 后端：下载授权校验、下载链接签发、文件不存在异常处理。

**验收标准**
- 已完成任务可下载对应交付文件。
- 非本人资源下载被拒绝。
- 下载失败可追踪错误原因。

**建议 PR 标题**
- `feat(client-delivery): complete downloadable delivery loop`

---

## [P0-SUPPORT-01] To B 最小支撑：不阻塞 Client 主链路

**目标**
- 仅保留机构侧必要接口稳定性，确保不影响 Client P0 主线。

**范围**
- 机构代下单与归属鉴权仅做缺陷修复与稳定性补齐。
- 不新增机构复杂业务（结算/报表等）。

**验收标准**
- 机构相关改动不影响 Client 登录与生成链路。
- 关键越权风险有自动化回归。

**建议 PR 标题**
- `fix(agency): stabilize minimum agency support for client p0`

## P0 验收门槛（强制）
- 必须提供以下验证命令与结果：
  - `pnpm --filter web lint`
  - `pnpm --filter web test`
  - `pnpm --filter web build`
  - `pnpm --filter api lint`
  - `pnpm --filter api test`
  - 关键链路 e2e（至少覆盖登录、题目、开题、正文）
- 任何只完成“页面壳子”的 PR 不得标记为 P0 完成。
